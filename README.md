# Shallow Water Equation (Kokkos)

This example ports the `swe.py` Runge–Kutta 4 pipeline to Kokkos. It reproduces
the gather/scatter stencil using portable parallel loops and can target OpenMP
or CUDA depending on which Kokkos build you link against.

---

## 1. Prerequisites

1. **Kokkos builds.** Build (or install) Kokkos twice—once for OpenMP only, once
   for CUDA Volta70. Example commands from the repo root (adjust compilers as
   needed):

   ```bash
   cd ~
   git clone https://github.com/kokkos/kokkos.git
   cd kokkos
   # OpenMP-only Kokkos
   cmake -B build-omp -S . \
     -DKokkos_ENABLE_OPENMP=ON \
     -DKokkos_ENABLE_CUDA=OFF \
     -DCMAKE_CXX_COMPILER=$(which g++)
   cmake --build build-omp -j

   # CUDA + OpenMP Kokkos (Volta70)
   cmake -B build-volta70 -S . \
     -DKokkos_ENABLE_OPENMP=ON \
     -DKokkos_ENABLE_CUDA=ON \
     -DKokkos_ARCH_VOLTA70=ON \
     -DCMAKE_CXX_COMPILER=$PWD/bin/nvcc_wrapper \
     -DCUDAToolkit_rt_LIBRARY=$CUDA_HOME/lib64/libcudart.so
   cmake --build build-volta70 -j
   ```

   Notes:
   - Setting `CUDAToolkit_rt_LIBRARY` to the shared `librt.so.1` avoids NVLink
     failures on systems where `/usr/lib/.../librt.a` is just a linker script.
   - Use the same compilers here that you intend to use for this example.

2. **Python + dependencies** (`numpy`, `h5py`) for exporting the data once.

---

## 2. Prepare binary input

The Python implementation loads two HDF5 files (`triangular_*.hdf5`,
`SW_*.hdf5`).  Convert them once to the simple binary layout consumed by the
Kokkos example:

> Requires Python 3 with `numpy` and `h5py` available on your PATH.

```bash
git clone https://github.com/jack-pan-ai/kokkos-benchmark.git
mv example/kokkos-benchmark example/shallow_water_equation 
bash example/shallow_water_equation/kokkos_swe_data.sh
```
Example:
```
python3 example/shallow_water_equation/swe_export_to_binary.py \
  ~/.easier/triangular_1000.hdf5 \
  ~/.easier/SW_1000.hdf5 \
  --output-dir data/swe_binary
```

Each output file stores `uint64` element counts followed by raw `int64` /
`float64` payloads. The driver infers sizes automatically.

## 3. Configure & build this example

This directory is an independent CMake project that links against the Kokkos
builds prepared above. `CMAKE_NO_SYSTEM_FROM_IMPORTED` is set internally so
standard headers come from your active toolchain (workaround for some HPC GCC
installs).

### 3.1 OpenMP-only build

```bash
cd example/shallow_water_equation
cmake -B build-omp -S . \
  -DKokkos_DIR=~/kokkos/build-omp/cmake_packages/Kokkos \
  -DCMAKE_CXX_COMPILER=$(which g++)
cmake --build build-omp --target shallow_water_pipeline -j
```

Point `Kokkos_DIR` at the CMake package directory generated by your OpenMP
Kokkos build/install (e.g., `build-omp/cmake_packages/Kokkos`).

### 3.2 CUDA Volta70 (+ OpenMP) build

```bash
(optional) cd example/shallow_water_equation
cmake -B build-volta70 -S . \
  -DKokkos_DIR=~/kokkos/build-volta70/cmake_packages/Kokkos \
  -DCMAKE_CXX_COMPILER=/home/x_panq/kokkos/bin/nvcc_wrapper \
  -DCUDAToolkit_rt_LIBRARY=$CUDA_HOME/lib64/libcudart.so
cmake --build build-volta70 --target shallow_water_pipeline -j
```

- `Kokkos_DIR` must reference the CUDA-enabled Kokkos config.
- The explicit `CUDAToolkit_rt_LIBRARY` mirrors the setting used when building
  Kokkos and ensures the shared `librt` is linked instead of the broken stub.

## 4. Run the simulation

### 4.1 OpenMP run

```bash
(optional)cd /home/x_panq/kokkos/example/shallow_water_equation
./build-omp/shallow_water_pipeline \
  --data ~/kokkos/data/swe_binary \
  --steps 1000 --dt 0.005 \
  --output swe_output_cpu --output-interval 10
```

Set `OMP_NUM_THREADS` to control host parallelism. Results appear in
`swe_output_cpu/data*.csv` if `--output` is specified.

### 4.2 CUDA run

```bash
cd /home/panq/dev/FlexSpmv/kokkos/example/shallow_water_equation
./build-volta70/shallow_water_pipeline \
  --data /home/panq/dev/FlexSpmv/kokkos/data/swe_binary \
  --steps 1000 --dt 0.0005 \
  --output swe_output_cuda --output-interval 10
```

Select GPUs via `CUDA_VISIBLE_DEVICES` as usual. When `--output` is set the code
writes `data000.csv`, `data001.csv`, … containing `(x, y, h)` triples.

## 5. Profiling runs

`bash example/shallow_water_equation/kokkos_swe.sh`

Add `--profile` to switch from RK4 stepping to the timing loop:

- Performs a warmup (`--profile-warmup`, default 10)
- Runs a timed loop (`--profile-iters`, default 20 on CPU, 200 on CUDA/HIP)
- Prints aggregate time + ms/iteration
- Appends CSV rows to `timing.csv` (or `<output>/timing.csv`)

### 5.1 OpenMP profiling example

  ```bash
  ./build-omp/shallow_water_pipeline \
    --data /home/panq/dev/FlexSpmv/kokkos/data/swe_binary \
    --profile --profile-warmup 20 --profile-iters 50 \
    --output swe_profile_cpu
  ```

- Outputs `exec_space,iterations,seconds,ms_per_iteration` into the CSV.

### 5.2 CUDA profiling example

  ```bash
  ./build-volta70/shallow_water_pipeline \
    --data /home/panq/dev/FlexSpmv/kokkos/data/swe_binary \
    --profile --profile-warmup 20 --profile-iters 100 \
    --output swe_profile_cuda
  ```

## 6. Troubleshooting tips

- **`math.h` / `stdlib.h` not found** when compiling with site-specific GCC:
  this project sets `CMAKE_NO_SYSTEM_FROM_IMPORTED=ON` so standard headers are
  taken from your active compiler instead of any `-isystem /usr/include` pushed
  by imported targets.

- **`nvlink fatal: Could not open ... librt.a`**: ensure both Kokkos and this
  example set `-DCUDAToolkit_rt_LIBRARY=/usr/lib/x86_64-linux-gnu/librt.so.1`
  (or the equivalent shared path on your system).

- **Mismatched CUDA architecture**: change `-DKokkos_ARCH_VOLTA70=ON` (and the
  app’s configure command) to fit your GPU, e.g., `AMPERE80`, `HOPPER90`, etc.

With those two environments in place the SWE executable builds and runs on both
OpenMP and CUDA backends without touching the core Kokkos source tree.

## 7. Sparse matrix-vector benchmark (GPU)

`shallow_water_spmv.cpp` is a GPU-focused port of `swe_spmv.py`. It reuses the
binary inputs generated in section 2 and maps the shallow-water connectivity
into a gather/scatter SpMV that accumulates into the height field. Only the CUDA
build is supported/tested.

### 7.1 Build the CUDA target

```bash
cd /home/panq/dev/FlexSpmv/kokkos/example/shallow_water_equation
cmake --build build-volta70 --target shallow_water_spmv -j
```

Ensure `build-volta70` was configured with a CUDA-enabled Kokkos (see section
3.2).

### 7.2 Run on the GPU

```bash
cd /home/panq/dev/FlexSpmv/kokkos/example/shallow_water_equation
./build-volta70/shallow_water_spmv \
  --data /home/panq/dev/FlexSpmv/kokkos/data/swe_binary \
  --warmup 20 --iterations 200
```

The executable runs a warmup, benchmarks the SpMV loop on the selected GPU, and
prints aggregate and per-iteration times. Use `CUDA_VISIBLE_DEVICES` to pick a
device, just like the full shallow-water pipeline.

Optional flag:

- `--save-csv <path>` writes a CSV containing the accumulated height vector and
  the most recent `A*x` result (default disabled).

### 7.3 cuSPARSE COO vs CSR benchmark

`shallow_water_cublas_spmv.cpp` loads the same binary inputs but exercises the
vendor library directly to time COO and CSR paths via `cusparseSpMV`, and
compares against the header-only CUB CSR SpMV implementation.

Build and run (CUDA-enabled toolchain required):

```bash
cmake --build build-volta70 --target shallow_water_cublas_spmv -j
./build-volta70/shallow_water_cublas_spmv \
  --data /home/panq/dev/FlexSpmv/kokkos/data/swe_binary \
  --warmup 20 --iterations 200
```

The executable prints the aggregate and per-iteration runtime for each format
(cuSPARSE COO, cuSPARSE CSR, CUB CSR when available). Adjust `--warmup` /
`--iterations` to taste. Set `CUDA_VISIBLE_DEVICES` to bind the test to a
specific GPU. If the build does not embed CUB kernels for the active GPU
architecture the run reports that the CUB benchmark was skipped.
